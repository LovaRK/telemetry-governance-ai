## 1. What is datasensAI?

datasensAI is a **Splunk Telemetry Intelligence Platform**. It connects to a Splunk Enterprise instance, analyzes every sourcetype and index in the environment, and answers three questions:

1. **What am I getting from this data?** (Is it being used? Is it valuable?)
2. **How much is it costing me?** (Storage, license, and ingestion costs)
3. **Where can I save money?** (Specific, actionable recommendations)

It combines **deterministic mathematical scoring** (formulas that always produce the same result for the same input) with **AI-powered narrative analysis** (a language model that writes human-readable explanations and recommendations).

**Key principle:** The numbers are never generated by AI. Every score, dollar amount, and percentage comes from a fixed mathematical formula. The AI only writes the explanation of what those numbers mean and what action to take.

---

## 2. The Problem It Solves

Organizations running Splunk typically have dozens or hundreds of data sources (called "sourcetypes" — for example, `firewall_logs`, `web_access`, `syslog`). Over time:

- Some sourcetypes become unused but keep costing money
- Retention policies may keep data longer than necessary
- Security detection rules may not cover all available data
- There is no single view of "which data is worth what I'm paying for it"

datasensAI provides that single view. Instead of manually auditing hundreds of sourcetypes, it automates the entire analysis in minutes and presents an executive dashboard showing exactly where money is being wasted and what to do about it.

## 4. Data Flow — Step by Step

### Step 1: Splunk Discovery

When you click "Refresh" on the dashboard, the application queries your Splunk instance's REST API to discover:

- All indexes and sourcetypes
- Daily ingestion volume (GB/day) per sourcetype
- Event counts and time spans
- Active alerts and correlation searches associated with each sourcetype
- Dashboard panels referencing each sourcetype
- Ad-hoc search activity by users
- Retention policies configured per index
- MITRE ATT&CK technique coverage (if Splunk Enterprise Security is installed)

This is read-only — datasensAI never writes to or modifies your Splunk instance.

### Step 2-5: Deterministic Scoring

The scoring engine processes each sourcetype through fixed mathematical formulas (detailed in Sections 5-8 below). This produces:

- Three sub-scores (Utilization, Detection, Quality) per sourcetype
- One composite score per sourcetype
- A tier classification per sourcetype
- A dollar savings estimate per sourcetype

### Step 6: AI Narrative

Each sourcetype is sent to the local AI model (Ollama gemma2:9b running on your laptop). The AI receives the raw data and deterministic scores, then generates:

- A written recommendation
- An action category (ARCHIVE, OPTIMIZE, ELIMINATE, RETAIN, or MONITOR)
- Reasoning for why this action is appropriate
- A confidence assessment

**Critical safeguard:** After the AI returns its response, the system overwrites any score the AI may have emitted with the deterministic score. This ensures the dashboard never shows AI-hallucinated numbers.

### Step 7: KPI Aggregation

All individual sourcetype results are aggregated into portfolio-level KPIs and stored in the PostgreSQL database. The dashboard reads from this database.

## 5. The Three Scoring Dimensions

Every sourcetype is evaluated on three independent dimensions. Each produces a score from 0 to 100.

### 5.1 Utilization Score (0-100)

**Question answered:** "How much is this data being used?"

**Why this matters:** Data that no one searches, no alert fires on, and no dashboard displays is costing you money for no return. The Utilization Score quantifies how actively each sourcetype is being consumed.

**How it is calculated:**

For each sourcetype, we count five usage signals and assign weights:

| Signal | Weight | Example |
|--------|--------|---------|
| Alert count (active alerts referencing this sourcetype) | ×3 | 5 alerts × 3 = 15 |
| Scheduled search count | ×3 | 2 searches × 3 = 6 |
| Dashboard panel count | ×2 | 3 panels × 2 = 6 |
| Ad-hoc search count (manual user searches) | ×1 | 10 searches × 1 = 10 |
| Distinct user count (unique users who searched) | ×2 | 4 users × 2 = 8 |

```
Weighted Sum = (alerts × 3) + (scheduled_searches × 3) + (dashboard_panels × 2)
             + (ad_hoc_searches × 1) + (distinct_users × 2)
```

The score is then **normalized relative to the most-used sourcetype** in your environment:

```
Utilization Score = (this sourcetype's weighted sum / highest weighted sum across all sourcetypes) × 100
```

**Why relative scoring?** Absolute thresholds would not work across different organizations. A sourcetype with 50 searches might be heavily used in a small environment but lightly used in a large one. Relative scoring ensures the "most used" always scores 100, and everything else is proportional.

**Why these weights?** Alerts (×3) and scheduled searches (×3) indicate operational dependency — the organization has built automation around this data. Dashboard panels (×2) and distinct users (×2) indicate human consumption. Ad-hoc searches (×1) are the weakest signal because they may be one-time investigations.

### 5.2 Detection Score (0-100)

**Question answered:** "How well is this data being used for security detection?"

**Why this matters:** Splunk is commonly used for security monitoring. A sourcetype that maps to many MITRE ATT&CK techniques but has few active alerts represents a **detection gap** — security potential that is not being realized.

**How it is calculated:**

```
MITRE Potential  = min(100, MITRE_technique_count × 1.25)
Lantern Potential = min(100, Lantern_usecase_count × 6.0)
Potential        = max(MITRE_potential, Lantern_potential)

Realized         = (active_alert_count / max_alert_count_across_environment) × 100

Detection Score  = (0.40 × Potential) + (0.60 × Realized)
```

**Special rule:** If a sourcetype has zero MITRE techniques AND zero Lantern use cases, the Detection Score is 0. This prevents non-security data (e.g., application logs) from getting an artificial detection score.

**Why 40/60 split?** Potential (what could be detected) matters less than Realized (what is actually being detected). The 60% weight on Realized rewards organizations that have operationalized their security coverage.

**What are MITRE ATT&CK techniques?** MITRE ATT&CK is an industry-standard catalog of known attack patterns (e.g., "Brute Force", "Phishing"). If a sourcetype's data could detect 20 different attack techniques, that's its MITRE technique count.

**What are Lantern use cases?** Lantern is Splunk's own catalog of use cases for each data source. A higher count means Splunk recognizes more ways to use this data for security.

**Detection Gap flag:** If a sourcetype maps to 15+ MITRE techniques but has alerts covering less than 25% of them, it is flagged as having a "Detection Gap" — significant security potential is going unrealized.

### 5.3 Quality Score (0-100)

**Question answered:** "How clean and well-formed is this data?"

**Why this matters:** Poorly formatted data (timestamp parsing errors, truncated events, field extraction failures) degrades search performance and produces unreliable results. High-quality data costs the same to store but produces better outcomes.

**How it is calculated:**

```
Approximate Events = daily_GB × 1,000,000 (events per GB estimate)
Issue Density      = weighted_issues / approximate_events
Quality Score      = max(0, 100 - (issue_density × 2000))
```

Where `weighted_issues` is the count of data quality issues found, with DateParser errors weighted at 0.5 (they're common and less severe) and all other issues at 1.0.

**Special case:** If daily volume is zero or negative, Quality Score = 100 (no data = no quality problems).

**Why this formula?** The score starts at 100 (perfect) and is penalized proportionally to the density of issues. A sourcetype with 1 issue per 10,000 events gets a minor penalty. A sourcetype with 1 issue per 100 events gets a major penalty. The ×2000 multiplier calibrates the scale so that typical issue densities produce meaningful score differences.

## 6. The Composite Score

**Question answered:** "Overall, how valuable is this sourcetype?"

**How it is calculated:**

```
Composite Score = (Utilization Weight × Utilization Score)
               + (Detection Weight × Detection Score)
               + (Quality Weight × Quality Score)
```

**Default weights:**

| Dimension | Default Weight | Why |
|-----------|---------------|-----|
| Utilization | 0.35 (35%) | Important but not dominant — low usage doesn't mean data is worthless |
| Detection | 0.40 (40%) | Highest weight because security coverage is critical for most Splunk deployments |
| Quality | 0.25 (25%) | Least weight because quality issues can be fixed; utilization and detection are more fundamental |

**These weights are configurable.** In Settings → Scoring Weights, you can adjust them. They must always sum to 1.00 (100%). The system validates this and blocks changes if they don't sum correctly.

**Pre-built profiles:**

| Profile | Util | Detect | Quality | When to use |
|---------|------|--------|---------|-------------|
| Balanced (default) | 0.35 | 0.40 | 0.25 | General-purpose environments |
| Security First | 0.25 | 0.50 | 0.25 | SOC-focused environments where detection coverage matters most |
| Operations First | 0.50 | 0.25 | 0.25 | IT Ops environments where operational use matters more than security |
| Data Quality | 0.30 | 0.30 | 0.40 | Environments with known data quality problems |

### Minimum Activity Gate

There is a safety mechanism to prevent "clean but useless" data from getting an artificially high score:

```
If Utilization < 2 AND Detection = 0:
    Composite Score is capped at 19.9 (forces Low-Value tier)
```

**Why?** A sourcetype with zero utilization and zero detection coverage could still have high quality (100), giving it a composite of 0.25 × 100 = 25, which would classify it as "Nice-to-Have". But data that nobody uses and has no security value is genuinely low-value regardless of its quality. The gate prevents this false positive.

## 7. Tier Classification

Based on the Composite Score, each sourcetype is classified into one of four tiers:

| Tier | Composite Score Range | What It Means | Recommended Action |
|------|----------------------|---------------|-------------------|
| **Critical** | 65 – 100 | Heavily used, high security value, or both. Essential to operations. | Protect and invest. Ensure high availability and full detection coverage. |
| **Important** | 40 – 64 | Meaningfully used or has significant detection potential. Worth keeping. | Maintain. Consider optimizing retention or search efficiency. |
| **Nice-to-Have** | 20 – 39 | Some use, but not operationally critical. Could be reduced. | Review. Consider archiving to cheaper storage or reducing retention. |
| **Low-Value** | 0 – 19 | Little to no usage, no security value, or flagged by the Minimum Activity Gate. | Candidate for elimination or aggressive archiving. |

**Why these thresholds?**

- 65 is the "Critical" boundary because it requires strong performance across multiple dimensions — you can't score 65 with just one high dimension
- 40 is the "Important" boundary because it represents meaningful but not outstanding value
- 20 is the "Nice-to-Have" boundary and aligns with the Minimum Activity Gate (19.9 cap)
- These thresholds were calibrated against the datasensAI calculation guide's reference data

## 8. Storage Savings Calculation

**Question answered:** "How much money can I save by optimizing storage?"

Storage savings are computed from three independent components:

### 8.1 Retention Savings

**Question:** "Am I keeping data longer than necessary?"

```
Excess Days          = max(0, current_retention_days - recommended_max_retention)
Retention Excess GB  = daily_average_GB × (excess_days / retention_days)
Monthly Storage Cost = cost_per_GB_per_day × 30
Retention Savings    = retention_excess_GB × monthly_storage_cost × 12 months
```

**Default values:** `recommended_max_retention = 365 days`, `cost_per_GB_per_day = $0.50`

**Example:** A sourcetype ingesting 2 GB/day with 730-day retention (but only 365 days needed):
- Excess = 730 - 365 = 365 days
- Excess GB = 2 × (365/730) = 1.0 GB
- Monthly cost = 0.50 × 30 = $15/GB/month
- Annual savings = 1.0 × $15 × 12 = **$180/year**

### 8.2 Compression Savings

**Question:** "Can I compress under-utilized data to save space?"

```
Low Utilization %        = max(0, 1 - utilization_score / 100)
Compression Opportunity  = daily_GB × low_utilization_% × compression_factor
Compression Savings      = compression_opportunity × monthly_storage_cost × 12 months
```

**Default values:** `compression_factor = 0.30` (assumes up to 30% of low-utilization volume can be compressed)

**Why tied to utilization?** Highly utilized data is frequently searched, so compressing it would slow queries. Low-utilization data is rarely accessed, making it a good candidate for compression.

### 8.3 Field Savings (Future Enhancement)

**Question:** "Am I indexing fields that nobody uses?"

```
Unused Fields    = fields_indexed - fields_used
Unused Field GB  = daily_GB × (unused_fields / fields_indexed)
Field Savings    = unused_field_GB × monthly_storage_cost × 12 months
```

**Current status:** This component is a placeholder (returns $0) because the field-usage lookup data from Splunk has not been wired yet. When implemented, it will identify indexes where many fields are being indexed but never searched, which wastes both storage and license capacity.

### Total Savings

```
Total Annual Savings = Retention Savings + Compression Savings + Field Savings
```

### Savings Confidence

The system assigns a confidence level to the savings estimate:

| Data Available | Confidence |
|---------------|------------|
| Retention + Compression + Field data | 95% |
| Retention + Compression (no field data) | 80% |
| Retention only | 60% |
| No data available | 0% |

Currently, with field data not yet wired, maximum confidence is 80%.

## 9. Executive KPIs Explained

The Executive Overview dashboard shows these portfolio-level KPIs:

### 9.1 ROI Score (0-100)

```
ROI Score = Average of all Composite Scores across all sourcetypes
```

**What it means:** A single number summarizing "how well is my Splunk investment performing?" An ROI of 70+ means most data is well-utilized and valuable. An ROI below 40 suggests significant waste.

**Why it works:** By averaging composite scores (which already factor in utilization, detection, and quality), the ROI captures overall portfolio health in one number.

### 9.2 GainScope (0-100%)

```
GainScope = (Total daily GB from Critical + Important tiers) / (Total daily GB across all tiers) × 100
```

**What it means:** "What percentage of your daily data ingestion is high-value?" A GainScope of 80% means 80% of what you're paying to ingest is genuinely valuable data. The remaining 20% is Nice-to-Have or Low-Value.

**Why it's useful:** Unlike ROI (which is count-based), GainScope is volume-weighted. One large low-value sourcetype ingesting 50 GB/day will significantly lower GainScope even if most sourcetypes are high-value.

### 9.3 Savings Potential ($)

```
Savings Potential = Sum of all deterministic storage savings across all sourcetypes
```

This is the total annual dollar savings achievable if all recommended optimizations were implemented.

### 9.4 Annual Spend ($)

```
Annual Spend = Sum of (daily_GB × cost_per_GB_per_day × 365) for all sourcetypes
```

The total annual cost of ingesting and storing all your Splunk data at the configured storage rate.

### 9.5 Low-Value Spend ($)

```
Low-Value Spend = Sum of annual cost for sourcetypes in "Nice-to-Have" or "Low-Value" tiers
```

How much of your annual spend goes toward data that scored below "Important". This is the primary target for cost optimization.

### 9.6 Critical Count

The number of sourcetypes classified as "Critical" tier. These are your most valuable data sources.

### 9.7 Low-Value Count

The number of sourcetypes classified as "Low-Value" tier. These are candidates for elimination or aggressive archiving.

## 11. AI Decision-Making Process

### What the AI Receives

For each sourcetype, the AI model receives:

- The sourcetype name and index
- Daily ingestion volume (GB/day)
- The three deterministic scores (Utilization, Detection, Quality)
- The composite score and tier classification
- Retention policy information
- Alert and search counts
- MITRE and Lantern coverage data

### What the AI Decides

The AI recommends one of five actions:

| Action | When Recommended | Expected Savings |
|--------|-----------------|------------------|
| **ARCHIVE** | Data has very low utilization and no security value. Move to cold/frozen storage. | 60% of annual license cost for that sourcetype |
| **OPTIMIZE** | Data is somewhat useful but could be reduced — e.g., excessive retention or unnecessary fields. | 20% of annual license cost |
| **ELIMINATE** | Data provides zero value and should be decommissioned entirely. | 100% of annual license cost |
| **RETAIN** | Data is valuable and should be kept as-is. | $0 (no change needed) |
| **MONITOR** | Data may have value but needs further observation before deciding. | $0 (no change yet) |

### How Confidence is Determined

The AI assigns a confidence level (HIGH, MEDIUM, or LOW). These are mapped to numeric values:

| AI Confidence | Numeric Value | What it means |
|--------------|---------------|---------------|
| HIGH | 0.90 | Strong evidence supports the recommendation |
| MEDIUM | 0.50 | Some evidence, but additional review recommended |
| LOW | 0.30 | Limited evidence — treat as preliminary |

### The Deterministic Override

**This is the most important safety feature of the system.**

After the AI generates its response, the system:
1. Extracts the action, reasoning, and narrative from the AI's response
2. **Discards any scores the AI may have generated**
3. **Overwrites them with the pre-computed deterministic scores**
4. Saves the combined result (deterministic numbers + AI narrative) to the database

This means:
- The numbers on the dashboard are **always** from the mathematical formulas
- The AI can never cause a score to be wrong
- The AI adds value through its narrative explanations and action recommendations, not through calculations

### Why Local AI?

The default AI model (Ollama gemma2:9b) runs entirely on your laptop. This means:

- **No data leaves your machine** — Splunk telemetry data, which may contain sensitive information, is processed locally
- **No API costs** — no per-request charges to an external AI service
- **Works offline** — after initial setup, the system works without internet
- **Reproducible** — the same model version produces consistent results

If your laptop cannot run the model (requires ~8 GB RAM), you can optionally switch to Anthropic's cloud API (claude-3-5-sonnet) in Settings → AI Provider. This requires an API key and sends data to Anthropic's servers.

## 12. The Governance Workflow

The Governance page provides human oversight over AI recommendations:

1. **Queue:** All AI recommendations appear in a review queue
2. **Review:** Each item shows the sourcetype, the AI's recommendation, reasoning, and confidence
3. **Approve/Reject:** An authorized user can approve or reject each recommendation
4. **Audit Trail:** Every decision is logged with who approved/rejected it and when
5. **Streaming:** Approved decisions are streamed to downstream systems for implementation

**Why governance?** AI recommendations should be reviewed by a human before acting on them. The governance workflow ensures no action is taken without explicit human approval, providing accountability and a complete audit trail.

## 15. Why This Approach?

### Why deterministic scoring instead of pure AI?

**Reproducibility.** If you run the pipeline twice on the same data, you get the same scores. AI models can produce different outputs for the same input (non-deterministic), so using them for scoring would make auditing impossible. By keeping scores deterministic, any analyst can verify the numbers independently using the formulas in this document.

### Why use AI at all?

**Narrative quality.** While the scores tell you "sourcetype X has a composite score of 23", the AI explains "sourcetype X is sending 5 GB/day of firewall connection logs that no one searches and no alerts fire on. Consider archiving to S3 to save $2,700/year." The AI turns numbers into actionable business language.

### Why three dimensions?

**Completeness.** A single score would hide important nuances. For example:
- High utilization + low detection = operationally important but a security gap
- Low utilization + high detection = security-critical but needs better tooling
- High quality + low utilization = well-formatted but unused data (waste)

Three dimensions let stakeholders see exactly where the value (or lack thereof) comes from.

### Why weighted composite instead of simple average?

**Flexibility.** Different organizations value different things. A SOC (Security Operations Center) cares more about detection coverage. An IT Ops team cares more about utilization. The weighted composite lets each team configure the scoring to match their priorities while keeping the individual dimension scores visible for context.

### Why a governance workflow?

**Accountability.** AI recommendations are suggestions, not commands. In enterprise environments, decommissioning a data source could have regulatory, compliance, or operational implications that the AI doesn't know about. The governance workflow ensures a knowledgeable human reviews every recommendation before it's acted upon.

### Why local-first AI?

**Data sovereignty.** Splunk telemetry metadata can reveal sensitive information about an organization's infrastructure, security posture, and operations. Running the AI locally ensures this data never leaves the organization's control. The Anthropic cloud option exists as a fallback for resource-constrained laptops, but it's opt-in and explicitly flagged.

## 17. Frequently Asked Questions

### Q: Why do my scores look different from the reference screenshots?

**A:** The reference screenshots were generated against a different Splunk environment (154 GB/day, 247 sourcetypes). Your environment has different data volumes, sourcetype counts, and usage patterns. The **formulas** are the same, but the **inputs** differ. This is expected — datasensAI adapts to whatever data exists in your Splunk.

### Q: Why is Low-Value Spend showing $0?

**A:** This means none of your sourcetypes scored below 20 (Low-Value tier) or below 40 (Nice-to-Have tier). If all your data is well-utilized and has security value, $0 Low-Value Spend is correct and is a positive indicator.

### Q: Why does the Quick Wins count change between pipeline runs?

**A:** Quick Wins are identified by the AI model. Each pipeline run may produce slightly different AI decisions because language models are non-deterministic. One run might flag 6 quick wins, the next might flag 1 or 4. The deterministic scores remain identical between runs — only the AI's narrative and action recommendations may vary.

### Q: Can I verify the numbers manually?

**A:** Yes. See Section 18 below. Every number on the dashboard can be independently computed using the formulas in this document and the raw data from your Splunk instance.

### Q: What if the AI recommends eliminating a sourcetype I know is important?

**A:** This is exactly why the Governance workflow exists. The AI makes recommendations based on quantitative signals (scores, usage patterns). It doesn't know about business requirements, compliance obligations, or future plans. Use the Governance page to reject inappropriate recommendations. The system learns nothing from your rejections — it's a safeguard, not a feedback loop.

### Q: How often should I run the pipeline?

**A:** Run it whenever you want a fresh analysis. There's no automatic schedule. Typical usage: run it once to get the initial picture, then re-run weekly or monthly to track changes. Each run replaces the previous results (they are not additive).

### Q: Does datasensAI modify my Splunk instance?

**A:** No. datasensAI only reads metadata from Splunk via the REST API. It never writes to, modifies, or deletes anything in your Splunk environment.

### Q: What happens if Ollama is not running when I start the pipeline?

**A:** The worker will fail to generate AI narratives and the pipeline will report an error. The deterministic scores will still be computed, but the AI recommendations will be missing. Start Ollama and run the pipeline again.

### Q: Can I change the storage cost rate?

**A:** Yes. In the Filter Bar at the top of the Executive Overview, you can adjust the "Storage $/GB/Month" parameter. Changing this will recalculate all savings estimates in real-time (no need to re-run the pipeline). The default is $15/GB/month.

## 18. Validation — How to Verify the Numbers

If your lead wants to verify that datasensAI's calculations match a manual analysis, here is the process:

### Step 1: Export Raw Data from Splunk

Run these Splunk searches to get the raw inputs:

```spl
| rest /services/data/indexes
  | stats sum(currentDBSizeMB) as totalMB by title
```

```spl
| tstats count where index=* by index, sourcetype
  | stats sum(count) as event_count dc(sourcetype) as sourcetype_count by index
```

### Step 2: Pick One Sourcetype and Compute Manually

Choose a sourcetype (e.g., `syslog`). Look up its values in the dashboard's Detail tab.

**Manual calculation example for `syslog`:**

1. **Utilization Score:** Count alerts (say 3), scheduled searches (2), dashboard panels (1), ad-hoc searches (5), distinct users (2):
   - Weighted sum = (3×3) + (2×3) + (1×2) + (5×1) + (2×2) = 9 + 6 + 2 + 5 + 4 = 26
   - If the highest weighted sum in the environment is 45:
   - Utilization = (26/45) × 100 = 57.8

2. **Detection Score:** If syslog maps to 20 MITRE techniques and has 5 active alerts, and the max alert count across the environment is 12:
   - MITRE potential = min(100, 20 × 1.25) = 25.0
   - Realized = (5/12) × 100 = 41.7
   - Detection = (0.40 × 25.0) + (0.60 × 41.7) = 10.0 + 25.0 = 35.0

3. **Quality Score:** If daily volume is 1 GB with 50 quality issues:
   - Approximate events = 1 × 1,000,000 = 1,000,000
   - Issue density = 50 / 1,000,000 = 0.00005
   - Quality = max(0, 100 - (0.00005 × 2000)) = 100 - 0.1 = 99.9

4. **Composite:** (0.35 × 57.8) + (0.40 × 35.0) + (0.25 × 99.9) = 20.23 + 14.0 + 24.98 = 59.2

5. **Tier:** 59.2 is between 40 and 64, so the tier is **Important**.

### Step 3: Compare

Compare your manually computed values against what the dashboard shows for the same sourcetype. They should match to within 0.1 (rounding differences).

If they don't match, the most common reasons are:
- Different time windows (the dashboard may be looking at a different date range)
- The "max weighted sum" normalization uses all sourcetypes, not just the one you picked
- Splunk REST API counts may differ from search-time counts
