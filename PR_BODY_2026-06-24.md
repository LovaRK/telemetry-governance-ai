## Migration

`apps/api/migrations/213_stage_event_heartbeat.sql` — single statement that
widens the `pipeline_stage_events.status` CHECK constraint to include
`'HEARTBEAT'`. Reversible (the DOWN would just narrow back). Idempotent
(uses `DROP CONSTRAINT IF EXISTS`).

```sql
ALTER TABLE pipeline_stage_events
  DROP CONSTRAINT IF EXISTS pipeline_stage_events_status_check;

ALTER TABLE pipeline_stage_events
  ADD CONSTRAINT pipeline_stage_events_status_check
    CHECK (status IN ('IN_PROGRESS', 'SUCCESS', 'FAILED', 'HEARTBEAT'));
```

## Assumptions / things reviewers should poke at

- The mock's `is1stmileVolumeLookup` handler uses **mock index names** (main, security, web_access…) as the carrier for the 92 GB proportions, not the real 1stmile index names (oswin, apptomcat, appapache…). That's correct because the same mock also returns mock index names from `/services/data/indexes` — but if you read only the mock's lookup response in isolation you might expect real 1stmile names. Worth a comment in code review.
- `ACCESS_TOKEN_TTL` raised to 1 h is a security trade-off: longer-lived access tokens are more exposed if leaked. Refresh tokens are still rotated on use and revoked, and we still single-flight refresh, so the effective window of a stolen access token is at most the TTL. For local desktop deploys this is fine; if anyone plans to deploy this multi-tenant on the open internet the TTL should be revisited.
- The `.gitignore` change is small but important — please don't squash-merge it out of the PR; future migrations rely on it being in main.
